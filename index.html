मूल पैटर्न — क्या टूट रहा है? The Core Pattern — What Is Breaking?

The system was designed for a world where supply is always available and partners always cooperate. That world doesn't exist. When supply fails, the system has no circuit breaker, no honest communication, and no manual override. Customers are left in a void of vague messages and unanswered calls until they give up. Structural Root Causes — सिस्टम में क्या टूटा है? Structural Root Causes — What Is Broken in the System?

हर root cause chain observable failure से शुरू होती है और "why" पूछते हुए system design decision तक पहुँचती है। Each chain starts from the observable failure and asks 'why' until it lands on a system design decision.

Root Cause A — "Thoda jaldi nahi ho sakta kya?" (Expectation Mismatch) — ~90 (~16%) Root Cause A — "Can't it be done faster?" (Expectation Mismatch) — ~90 (~16%) ▼
OBSERVABLE FAILURE
~90 customers (~16%) cancelled within 12h with zero system failures. They simply expected the install to happen faster than the system delivered.
WHY?
Because installation did not happen within the speed they expected.
WHY?
Because their mental model is same-day or within-hours activation, benchmarked against Jio, Airtel and local cable.
WHY?
Because the booking journey did not anchor the expected install TAT upfront.
WHY?
Because the expected installation timeline is not being communicated enough or repeatedly.
STRUCTURAL ROOT CAUSE

Pre-booking expectation setting on installation TAT is weak. The system takes a booking without ever forcing the customer to confirm they understand and accept the install TAT.

Root Cause C — "Bohot der se wait kar rahe hai aur kuch nahi ho raha hai" (Accumulated System Failure) — ~260 (~45%) Root Cause C — "Been waiting forever, nothing is happening" (Accumulated System Failure) — ~260 (~45%) ▼
OBSERVABLE FAILURE
127 customers (22%) cancelled after 2+ failures. 9 customers experienced 5+ failures with an average wait of 258 hours. After each failure they were routed back into the same scheduling loop.
WHY?
Because repeated failed attempts exhausted trust and patience.
WHY?
Because after each failure the customer re-entered the same scheduling loop.
WHY?
Because failed cases are not differentiated from first-time installs.
WHY?
Because the workflow is optimised for standard installs, not failure recovery.
STRUCTURAL ROOT CAUSE

Repeated-failure customers are routed back into BAU instead of a recovery workflow. The system has no notion of a 'failed case' that needs different handling.

Root Cause D — "Koi madad nahi kar paa raha" (Fragmented Information Architecture) Root Cause D — "Nobody can help me" (Fragmented Information Architecture) ▼
OBSERVABLE FAILURE
412 (71%) called CC at least once. 183 (32%) were heavy callers (5+ calls). One customer called 47 times. CC save rate: 0%.
WHY?
Because they could not understand what was happening with their booking.
WHY?
Because they did not receive a clear and reliable status update.
WHY?
Because CC, bot and technician do not reflect the same piece of information at the same time.
WHY?
Because partner-side status updates and execution milestones are not being synced to customer-facing systems in real time.
STRUCTURAL ROOT CAUSE

There is no single source of truth for the customer's install state. Every channel — chatbot, CC, tech — sees a partial view, so the customer cannot get a reliable answer from anyone.

खंड 7 SECTION 7

CX Call Learnings — क्या सुनाई दिया calls में CX Call Learnings — What We Heard in Calls

50+ CC, partner & technician calls सुनकर ये patterns निकले: After listening to 50+ CC, partner & technician calls, these patterns emerged:

Customer frustration: He couldn't figure out if he made the payment or not. And since he was not getting any call so he felt "kuch toh gadbad hai"
Technician unreachable: Customer couldn't get through the technician and wanted to request faster installation. 3 times call was transferred. CX need to ensure they call on the technician number.
Price shock: "You have to pay 100 then 300 then 2 day free and then plan amount — what is this? Why so? Why should I pay so much."
Heavy caller (9 calls, 4 days waiting): He kept getting dates and CX says "jo bhi chat mein likha hai wahi hai." Every call they feel the next date is for real but when it is not honored then their trust is broken.
Heavy caller (7 days, no setup): Setup date was 22nd and then it was not. At CX we tell on CC that it is delayed and new date will be confirmed. No answer or clarity again. Same loop. He eventually asked to cancel.
Heavy caller (technician not picking up): "Maine 10 times kar diya hai download nothing is happening." We say "I'm checking the status" and then the same thing.
Auto-refund trigger: "Give me a date by when this will happen" — the moment the CX said "you will get auto refund" it seemed he was convinced of cancelling.

कार्यप्रणाली — हमने ये कैसे किया Methodology — How We Did This

1
Chat Journey Analysis: All 578 message journeys classified by pain point themes using keyword + pattern matching across 7 theme categories
2
Stated Reason Classification: All cancellation reasons (Hindi/Hinglish) translated and grouped into 13 theme categories
3
Interaction Pattern Profiling: Customers segmented by CX calling behavior, tech interaction, failure count, and cancellation speed
4
5-Why Root Cause Analysis: Each pain point traced to system-level root causes through iterative "why" questioning with data evidence at each level
5
Root Cause Sizing: Each root cause quantified by affected customers, with overlap matrix showing interconnections
6
Listening to 50+ CC and Partner CX calls.
